\chapter{Manual técnico}
\label{ap:manual-tecnico}

Este manual recoge la información operativa necesaria para desplegar, probar y
mantener la segunda fase del laboratorio. No sustituye a los capítulos de
diseño y evaluación: sirve de guía de referencia para reproducir una
instalación de desarrollo, diagnosticar fallos habituales y extender el sistema.

\section{Estructura relevante}

\begin{verbatim}

\section{Dependencias de ejecución}

La ejecución completa requiere cuatro servicios compartidos: PostgreSQL para
metadatos, MinIO para artefactos, Neo4j para el grafo y Qdrant para las memorias
indexadas. El acceso a modelos de lenguaje se realiza mediante OpenRouter.
Voyage AI y ZeroEntropy son dependencias opcionales de la capa semántica: si no
están configuradas, el flujo principal sigue disponible, pero no hay
recuperación semántica, \textit{reranking} ni escritura indexada de memorias de
simulación en Qdrant.

\section{Configuración}

La configuración vive en \texttt{phase2-juan/.env}. Las variables principales
son las conexiones a PostgreSQL, MinIO, Neo4j y Qdrant, la clave
\texttt{OPENROUTER\_API\_KEY} y varios conmutadores de capacidades opcionales:

\begin{itemize}
\item \texttt{ENABLE\_KNOWLEDGE\_READ}: activa la lectura del Knowledge Backbone.
\item \texttt{ENABLE\_KNOWLEDGE\_WRITE}: activa la escritura de memorias de simulación.
\item \texttt{ENABLE\_QUERY\_HISTORY}: activa la consulta NL-SQL del historial.
\item \texttt{ENABLE\_CHAT\_PERSISTENCE}: activa la persistencia del chat.
\end{itemize}

Con esta configuración, una instalación de desarrollo puede mantener operativo
el recorrido básico aunque algunas capacidades opcionales, como la consulta
NL-SQL o la escritura de memorias, estén desactivadas. El archivo
\texttt{.env.example} contiene la lista completa de variables esperadas.

\section{Comandos}

\begin{verbatim}
# Infraestructura para levantar los contenedores
docker compose up -d

# Backend
cd phase2-juan
uv sync
uv run uvicorn simlab.api:app --port 8000

# Frontend
cd phase2-juan/frontend
npm install
npm run dev

# CLI
cd phase2-juan && uv run simlab
\end{verbatim}

El orden recomendado de arranque es: primero los servicios compartidos,
después el backend y por último el cliente web. Si el backend se inicia antes
de que PostgreSQL o MinIO estén disponibles, las primeras consultas pueden
fallar aunque el proceso de FastAPI siga vivo. En ese caso basta con comprobar
el estado de los contenedores y reiniciar el backend.

\section{Flujo operativo de una sesión}

Desde el punto de vista de operación, una sesión completa sigue este recorrido:

\begin{enumerate}
\item El cliente abre el WebSocket de \texttt{simlab.api} y envía el mensaje
del usuario al Orchestrator.
\item El Orchestrator consulta los modelos disponibles mediante
\texttt{model\_loader} y presenta al usuario las opciones cargadas desde la
primera fase.
\item Cuando el usuario confirma, el Architect genera una especificación de
entorno que se valida con los modelos de \texttt{simlab.spec}.
\item \texttt{Environment} ejecuta los modelos paso a paso. Cada decisión se
registra como \texttt{Event} y se publica al frontend para alimentar el
\textit{replay}.
\item El Tracker transforma la ejecución en trayectorias, episodios y eventos
críticos.
\item El Analyst interpreta esos datos, genera métricas y gráficas, y puede
consultar el Knowledge Backbone mediante \texttt{retrieve\_context}.
\item El Reporter compone el cuerpo LaTeX del informe y lo compila a PDF
mediante la plantilla del laboratorio.
\item El Orchestrator persiste conversación, ejecución, artefactos y
observaciones de simulación.
\end{enumerate}

\section{Persistencia y artefactos}

PostgreSQL guarda los metadatos principales: experimentos, ejecuciones,
mensajes de chat, modelos registrados, artefactos y observaciones de
simulación. MinIO conserva los binarios, principalmente modelos Python de la
primera fase e informes PDF de esta fase. Qdrant almacena las memorias densas y
\textit{sparse}; Neo4j mantiene el grafo de paradigmas, formulaciones, modelos
y procedencia científica cuando esos datos existen.

La conversación no queda solo como estado de interfaz. Los mensajes se
persisten para auditoría y para que el Orchestrator pueda compactar el
historial sin perder las decisiones tomadas. La autocompactación sustituye
turnos antiguos por un resumen determinista cuando el contexto crece demasiado.

\section{Pruebas}

\begin{verbatim}
cd phase2-juan && uv run pytest tests/ -v
cd phase2-juan/frontend && npx playwright test
\end{verbatim}

Las pruebas con servicios reales requieren que \texttt{docker compose} esté
levantado. Las pruebas con respuestas simuladas validan el
pipeline y la interfaz aunque no haya proveedores externos configurados. Para
cambios en la interfaz se recomienda complementar Playwright con una revisión
manual en \texttt{http://localhost:5173}, porque la suite comprueba presencia y
flujo, pero no sustituye la inspección visual.

\section{Mantenimiento y extensión}

Un nuevo modelo de decisión generado por la primera fase queda disponible para
esta fase sin necesidad de tocar el código del laboratorio: la primera fase lo
registra en la tabla \texttt{models} de PostgreSQL y sube el archivo Python a
MinIO; el \texttt{model\_loader} de \texttt{simlab/model\_loader.py} descubre la
nueva entrada en la siguiente sesión y carga el archivo desde MinIO al
instanciar el modelo.

Para añadir una nueva herramienta al Orchestrator se registra su esquema en el
\textit{tool registry} de \texttt{simlab/orchestrator.py} junto con su
implementación. Para añadir un agente especializado se sigue el patrón de los
existentes: crear un módulo en \texttt{simlab/}, definir su entrada y salida
esperadas y registrar su \textit{tool call} en el Orchestrator. Los cambios
sobre la capa de persistencia compartida se incorporan mediante migraciones
Alembic en \texttt{shared/migrations/versions/}.
