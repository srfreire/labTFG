\chapter{Introducción}

\section{Contexto y alcance}

Este Trabajo de Fin de Grado forma parte de un laboratorio virtual para
simular, observar y analizar paradigmas de toma de decisiones humanas mediante
agentes inteligentes. El proyecto completo se divide en 2 partes: esta memoria
documenta la generación automática de modelos de decisión, y la parte de Juan
Freire Álvarez documenta la simulación, observación, análisis y reporte de esos
modelos.

La frontera entre ambas partes se fija con un contrato común. La fase 1 produce
modelos Python autocontenidos, artefactos revisables y metadatos persistentes.
La fase 2 consume los modelos aprobados y los ejecuta dentro del laboratorio. La
memoria se centra en la fase 1, aunque menciona la infraestructura compartida
necesaria para persistencia, trazabilidad y memoria.

\section{Problema y motivación}

El laboratorio necesita modelos de decisión que actúen dentro de entornos de
simulación. Crear esos modelos a mano es costoso porque antes de escribir código
hay que identificar paradigmas científicos, traducir postulados a variables y
ecuaciones, adaptar la formulación a acciones y percepciones concretas, y
comprobar que el resultado cumple un contrato ejecutable.

El problema principal es la distancia entre una descripción en lenguaje natural
y un modelo computacional verificable. Una petición como ``simular toma de
decisiones basada en regulación homeostática'' no contiene autores,
parámetros, variables internas, reglas de actualización, mapeo al entorno ni
pruebas. Si cada nuevo paradigma exige repetir todo ese trabajo de forma manual,
el laboratorio deja de ser escalable.

La motivación de esta fase es construir una cadena trazable entre conocimiento
científico y software ejecutable. El sistema no sustituye el criterio humano:
produce artefactos intermedios para que una persona revise paradigmas,
formulaciones, specs y modelos antes de aceptarlos. Esta trazabilidad es
importante en modelado computacional de conducta, donde un modelo puede pasar
tests y aun así representar mal el fenómeno estudiado \cite{wilson2019modeling}.

\section{Objetivos}

El objetivo general es construir un sistema capaz de transformar una descripción
de un problema de decisión en modelos Python ejecutables, probados y
preparados para el laboratorio virtual.

\section{Descripción del sistema y metodología de desarrollo}

El sistema se organiza como un pipeline multiagente. Un Router determinista
coordina etapas especializadas: clasificación, investigación, formalización,
adaptación al entorno, construcción, validación y escritura de memoria. Cada
etapa produce artefactos persistentes que pueden revisarse antes de continuar.

La salida final cumple el contrato usado por el laboratorio:

\begin{lstlisting}[language=Python]
def decide(self, perception: dict) -> Action
def update(self, action, reward, new_perception) -> None
def get_state(self) -> dict
\end{lstlisting}

El desarrollo del propio sistema también se hizo con apoyo intensivo de agentes
de programación. Codex y Claude Code ayudaron a implementar, probar y
refactorizar, pero las decisiones de arquitectura, alcance, aceptación y
validación quedaron bajo supervisión humana. El capítulo 4 describe esa forma
de trabajo.

\section{Estado del arte}

El estado del arte relevante se agrupa en 2 bloques. El primero son sistemas y
entornos que permiten estudiar comportamiento mediante simulación. En este
grupo entran los laboratorios virtuales, los entornos de agentes y los modelos
computacionales de decisión: aprendizaje por refuerzo \cite{sutton2018rl},
teoría prospectiva \cite{kahneman1979prospect}, racionalidad limitada
\cite{simon1955behavioral} y modelos de difusión \cite{ratcliff1978theory}.
Estos trabajos aportan mecanismos formales, pero suelen requerir trabajo
experto para convertir literatura en modelos ejecutables.

El segundo bloque son tecnologías de Agentic AI. Los agentes con herramientas,
como ReAct y Toolformer, combinan razonamiento y acción
\cite{yao2023react,schick2023toolformer}. Los sistemas multiagente, como
AutoGen y ChatDev, dividen tareas largas en roles especializados
\cite{wu2023autogen,qian2024chatdev}. La recuperación aumentada y los grafos de
conocimiento permiten usar memoria persistente y contexto externo
\cite{lewis2020rag,edge2024graphrag}. Los agentes de programación, como
SWE-agent, muestran que la generación de código mejora cuando existe una
interfaz de herramientas y validación ejecutable \cite{yang2024sweagent}.

Este trabajo combina ambos bloques. Frente a un entorno de simulación
tradicional, automatiza parte de la generación de modelos. Frente a un agente de
programación convencional, no parte de un issue técnico, sino de una pregunta
científica que debe investigarse, formalizarse y validar antes de convertirse
en código.

\section{Estructura de la memoria}

La memoria sigue una estructura de TFG de desarrollo software. El capítulo 2
presenta requisitos. El capítulo 3 describe el diseño. El capítulo 4 explica el
desarrollo con IA y las decisiones técnicas. El capítulo 5 resume pruebas y
evals. El capítulo 6 recoge conclusiones y ampliaciones. Los anexos contienen
manual técnico, manual de usuario y material extendido de diseño, desarrollo y
evaluación.
