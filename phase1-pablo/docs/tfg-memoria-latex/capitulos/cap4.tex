\subsection{Integración continua y flujo de Git}

El repositorio incluye un flujo de GitHub Actions llamado \texttt{CI}. Se
ejecuta en cada \textit{push} a \texttt{main}, en cada \textit{pull request}
contra \texttt{main} y también de forma manual mediante
\texttt{workflow\_dispatch}. Además, el flujo cancela ejecuciones antiguas de
la misma rama cuando llega un \textit{push} nuevo, para que la señal de CI
corresponda siempre al último commit.

La validación automática tiene cuatro trabajos:

\begin{itemize}
\item \texttt{Lint + format}: instala \texttt{ruff} con \texttt{uv} y ejecuta
el linter y el check de formato desde la raíz del repositorio;
\item \texttt{Tests - shared}: sincroniza dependencias de \texttt{shared} con
\texttt{uv sync} en modo frozen, incluyendo extras de desarrollo, y ejecuta los
tests unitarios;
\item \texttt{Tests - phase1}: sincroniza \texttt{phase1-pablo} y ejecuta sus
tests unitarios;
\item \texttt{Tests - phase2}: sincroniza \texttt{phase2-juan} y ejecuta sus
tests unitarios.
\end{itemize}

Los jobs de pruebas excluyen marcas de integración, e2e y \texttt{real\_llm},
porque esas suites dependen de servicios externos o de ejecuciones costosas. El
flujo diario queda así dividido en dos capas: en cada \textit{push} se ejecuta
una compuerta rápida de formato, lint y tests unitarios; para cambios de mayor
riesgo se lanzan evals online o pruebas de integración de forma explícita.

\section{Evals y benchmarks}

Además de las pruebas unitarias, el proyecto incluye un arnés de evaluación
específico para el sistema. Esta parte es importante porque un pipeline basado
en agentes no se valida solo comprobando que una función devuelve el tipo
esperado. También hay que comprobar que los agentes se coordinan, que las
herramientas se invocan cuando corresponde, que la memoria crece sin degradarse
y que el sistema completo sigue funcionando después de cambios en prompts,
modelos o infraestructura. Esta necesidad es coherente con trabajos sobre
agentes de software y sistemas multiagente, donde la evaluación debe cubrir el
comportamiento del sistema completo y no solo llamadas aisladas
\cite{qian2024chatdev,yang2024sweagent}.

Las evals se definen como suites YAML en \texttt{evals/suites/}. Cada suite
declara las etapas que debe ejecutar, los temas de entrada, las aprobaciones
automáticas necesarias, el presupuesto máximo y las aserciones esperadas. Se
ejecutan desde la CLI:

\begin{verbatim}
uv run decisionlab eval run evals/suites/smoke.yaml
uv run decisionlab eval run evals/suites/full-pipeline.yaml
\end{verbatim}

El resultado de cada suite se guarda en:

\begin{verbatim}
evals/reports/<fecha>-<suite>/report.md
evals/reports/<fecha>-<suite>/report.json
\end{verbatim}

El informe incluye estado global, temas ejecutados, aserciones superadas o
fallidas, crecimiento del grafo, escrituras de memoria, duración por etapa,
latencia de herramientas y coste estimado. Esto permite usar las evals como
benchmark de regresión: si un cambio mejora un prompt pero empeora la
canonicalización, la recuperación o la latencia, queda reflejado en el reporte.

\subsection{Evals del sistema multiagente}

Estas evals prueban el flujo de agentes como sistema. Cubren Router, estados,
subagentes, herramientas, artefactos, aprobación automática en modo eval y salida
final. Su objetivo no es evaluar una memoria concreta. Comprueban que el pipeline
se mueve de una etapa a otra y que los artefactos generados son utilizables por
la siguiente etapa.

\begin{table}[H]
\begin{center}
\small
\begin{tabular}{|p{3.2cm}|p{2.5cm}|p{6.6cm}|} \hline
\textbf{Suite} & \textbf{Etapas} & \textbf{Qué valida} \\ \hline
Smoke & Research & Arranque básico del pipeline, éxito de una ejecución y descubrimiento mínimo de paradigmas. \\ \hline
Full pipeline & Research--Build & Ejecución completa: investigación, formalización, razonamiento, generación de código, importación del modelo y llamada funcional a \texttt{decide}. \\ \hline
\end{tabular}
\caption{Evals del flujo multiagente.}
\end{center}
\end{table}

Las aserciones de este bloque miran la salida del pipeline y el comportamiento
de los agentes:

\begin{verbatim}
paradigm, min_paradigms, tool_called,
has_formulation, module_imports,
decide_returns_action
\end{verbatim}

Así se cubren fallos que no aparecerían en una prueba unitaria aislada. Por
ejemplo, una prueba puede verificar que el Router avanza de estado, pero la eval
completa comprueba que Researcher, Formalizer, Reasoner y Builder producen
artefactos compatibles entre sí. Del mismo modo, importar el módulo generado y
llamar a \texttt{decide} ofrece una evidencia ejecutable mínima de que el
modelo final cumple el contrato esperado.

\subsection{Evals del sistema de memoria}

El segundo grupo de evals se centra en el sistema de memoria y conocimiento:
Qdrant, Neo4j, PostgreSQL, canonicalización, recuperación, deduplicación,
crecimiento del grafo y validez temporal. Estas pruebas son distintas porque
evalúan comportamiento acumulativo entre ejecuciones. No basta con que un agente
responda bien una vez; la memoria debe seguir siendo útil después de varios runs
y no debe introducir duplicados, hechos obsoletos o recuperaciones irrelevantes.
Por eso se evalúan señales propias de recuperación aumentada, memoria a largo
plazo y grafos de conocimiento
\cite{lewis2020rag,zhong2023memorybank,packer2023memgpt,hogan2021knowledgegraphs,edge2024graphrag}.

\begin{table}[H]
\begin{center}
\small
\begin{tabular}{|p{3.2cm}|p{2.5cm}|p{6.6cm}|} \hline
\textbf{Suite} & \textbf{Etapas} & \textbf{Qué valida} \\ \hline
Cumulative growth & Research & Crecimiento acumulativo del grafo en temas relacionados, reutilización de slugs canónicos y control de duplicados. \\ \hline
Canonicalization & Research & Uso de memoria, reutilización de paradigmas conocidos y detección de fugas de slugs no canónicos. \\ \hline
Slug accuracy & Research & Tasa de acierto contra un oráculo de slugs, crecimiento por etiqueta y latencia p95 de recuperación. \\ \hline
Memory retrieval & Research & Recuperación de paradigmas ya presentes en memoria y conservación de relaciones del grafo. \\ \hline
Retrieval quality & Offline & Calidad directa de recuperación: hechos sembrados deben aparecer en el top-k para consultas semánticas y léxicas. \\ \hline
Temporal correctness & Offline & Correctitud temporal de memorias: validez por fecha y cadenas de supersesión. \\ \hline
\end{tabular}
\caption{Evals del sistema de memoria y conocimiento.}
\end{center}
\end{table}

Las aserciones de memoria combinan comprobaciones directas contra
infraestructura y métricas agregadas de suite:

\begin{verbatim}
slug_hit_rate, kg_growth_rate, p95_below,
memory_at_time, supersession_chain_length,
retrieval_finds
\end{verbatim}

Esto cubre fallos propios de un sistema de memoria. Una prueba puede verificar
que \texttt{retrieve\_knowledge} se llama, pero una eval de calidad comprueba
que devuelve el hecho correcto. Una prueba puede comprobar que un nodo se
escribe en Neo4j, pero una eval de crecimiento detecta si el sistema empieza a
crear slugs duplicados para paradigmas ya conocidos. Del mismo modo, la suite
temporal valida que una memoria supersedida deja de aparecer en consultas
posteriores, algo que no puede comprobarse mirando solo el último estado del
grafo.

En la suite \texttt{full-pipeline}, la ejecución representativa recorrió

El último caso de la tabla también es útil: se conserva como una alerta

\texttt{slug-accuracy} detectó que dos temas relacionados con aprendizaje por

El arnés también incorpora límites de presupuesto. Cada suite puede declarar un
coste máximo estimado; si se supera, la ejecución se corta y el reporte deja
constancia de ello. Esta restricción es necesaria porque las evals ejecutan
agentes reales, búsquedas, extracción de memoria y llamadas a modelos. En la
práctica, las suites baratas se usan como smoke tests frecuentes y las suites
caras, como \texttt{full-pipeline.yaml}, se reservan para regresiones
importantes.

El propio arnés cuenta con pruebas automáticas para el parseo de suites YAML, la
ejecución de etapas, las aserciones, el cálculo de tiempos, los límites de
presupuesto y la generación de reportes. Esto evita que una eval falle por un
error del marco de evaluación en lugar de por una regresión del sistema evaluado.

\subsection{Evaluación con corpus cerrado de papers}
\label{sec:evaluacion-corpus-cerrado}

La evaluación más exigente de la fase 1 se hizo con 2 casos basados en PDFs. En
cada caso, el sistema solo podía acceder a los papers incluidos en el ZIP de
entrada. Para simular una búsqueda web controlada, las herramientas
\texttt{web\_search} y \texttt{search\_papers} devolvían metadatos y snippets
de esos documentos. Cuando el pipeline necesitaba leer un paper, la herramienta
de papers devolvía el texto extraído completo.

Esta configuración mide algo distinto a una ejecución normal. El objetivo no
era pedir un paradigma concreto, sino comprobar si el pipeline podía descubrir
los paradigmas relevantes, proponer formulaciones, razonar sobre un entorno,
generar modelos y conservar todos los artefactos usando solo un corpus cerrado.
Después se usó un LLM como juez, con el mismo prompt y la misma rúbrica para los
2 casos. El juez no participó en la generación de los artefactos. Su papel fue
revisar el resultado como un evaluador externo: fidelidad al corpus, calidad de
paradigmas, formulaciones, modelos, memoria, trazabilidad y advertencias.

Esta eval amplía la validación técnica porque separa 2 preguntas. La primera es
si el sistema termina una ejecución completa y genera código que cumple el
contrato \texttt{DecisionModel}. La segunda es si los artefactos generados son
coherentes con los papers de entrada y pueden ser revisados por un experto. La
primera pregunta se cubre con tests y estado \texttt{PASS}; la segunda se cubre
con el juez y con el bundle auditable de cada caso.

\section{Pruebas técnicas}

\subsection{Pruebas del Builder}

El Builder genera tanto el modelo como sus pruebas. Después ejecuta las pruebas
mediante:

\begin{verbatim}
uv run pytest
\end{verbatim}

El objetivo es verificar que el modelo respeta el contrato esperado:

\begin{itemize}
\item \texttt{decide} devuelve una acción válida;
\item \texttt{update} modifica el estado después de recibir recompensa;
\item \texttt{get\_state} expone variables internas serializables;
\item el modelo puede ejecutarse con percepciones del entorno.
\end{itemize}

La validación del sistema debe leerse con cautela. Las pruebas y evals muestran
que el pipeline funciona y produce artefactos auditables, pero no convierten cada
modelo generado en una validación científica definitiva.
