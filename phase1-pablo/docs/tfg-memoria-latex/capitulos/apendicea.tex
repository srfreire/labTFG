\chapter{Manual técnico}

Este anexo resume la instalación, estructura y operación técnica de la fase 1.
No documenta la implementación interna de la fase de simulación; solo menciona
la infraestructura compartida que la fase 1 utiliza para persistencia,
artefactos y memoria.

\section{Servicios requeridos y opcionales}

La fase 1 distingue entre servicios necesarios para una ejecución normal y
servicios opcionales que enriquecen la memoria.

\begin{table}[H]
\begin{center}
\small
\begin{tabular}{|p{3cm}|p{3cm}|p{6.3cm}|} \hline
\textbf{Servicio} & \textbf{Carácter} & \textbf{Uso en fase 1} \\ \hline
PostgreSQL & Requerido & Metadatos de runs, artefactos, modelos registrados y ciclo de vida de memorias. \\ \hline
MinIO & Requerido & Almacenamiento S3-compatible de informes, specs, código, tests, estado y trazas. \\ \hline
Neo4j & Opcional & Grafo de conocimiento con paradigmas, papers, variables, postulados, formulaciones y modelos. \\ \hline
Qdrant & Opcional & Recuperación de hechos mediante embeddings densos y BM25 nativo. \\ \hline
Voyage AI & Opcional & Embeddings de documentos y consultas. \\ \hline
ZeroEntropy & Opcional & Reranking de resultados recuperados. \\ \hline
\end{tabular}
\caption{Servicios usados por la fase 1.}
\end{center}
\end{table}

El contenedor \texttt{Services} encapsula estas dependencias. PostgreSQL y MinIO
son necesarios para persistir estado y artefactos. Neo4j, Qdrant y embeddings
pueden no estar disponibles; en ese caso el pipeline se ejecuta en modo
degradado, sin memoria avanzada ni recuperación aumentada.

\section{Variables de entorno}

Las variables principales se leen desde \texttt{shared/shared/settings.py}. En
desarrollo existen valores por defecto para servicios locales:

\begin{verbatim}
MINIO_ENDPOINT=localhost:9000
MINIO_ACCESS_KEY=minioadmin
MINIO_SECRET_KEY=minioadmin
MINIO_BUCKET=labtfg
POSTGRES_DSN=postgresql+asyncpg://.../labtfg
NEO4J_URI=bolt://localhost:7687
NEO4J_USER=neo4j
NEO4J_PASSWORD=labtfg00
QDRANT_URL=http://localhost:6333
VOYAGE_API_KEY=
ZEROENTROPY_API_KEY=
\end{verbatim}

La configuración de modelos de lenguaje se gestiona desde la configuración de
\texttt{decisionlab}. En despliegue se usan variables de entorno del proveedor
correspondiente, como Anthropic u OpenRouter.

\section{Arranque de infraestructura}

Desde la raíz del repositorio, el stack común se levanta con:

\begin{verbatim}
docker compose up -d postgres minio neo4j qdrant
\end{verbatim}

Para levantar también el servidor de fase 1:

\begin{verbatim}
docker compose up -d phase1-server
\end{verbatim}

El \texttt{docker-compose.yml} de la raíz define PostgreSQL, MinIO, Neo4j,
Qdrant y el servidor de fase 1. La fase 1 recibe las URLs y credenciales de esos
servicios por variables de entorno.

\section{Ejecución desde CLI}

El proyecto usa \texttt{uv} para Python. Desde \texttt{phase1-pablo/}, los
comandos habituales son:

\begin{verbatim}
uv run decisionlab --help
uv run decisionlab pipeline "Problema de toma de decisiones"
uv run decisionlab eval run evals/suites/smoke.yaml
uv run decisionlab eval run evals/suites/full-pipeline.yaml
\end{verbatim}

Las suites caras ejecutan agentes reales y pueden consumir presupuesto de
proveedor. Las suites offline validan propiedades de memoria sin llamadas a
modelos.

\section{Artefactos generados}

La fase 1 genera artefactos por ejecución. En una ejecución real se almacenan en
MinIO con prefijos asociados al \texttt{run\_id}; en la ejecución de ejemplo del
repositorio aparecen materializados en \texttt{examples/sample-run/}.

\begin{verbatim}
report.md
deep/<paradigm>.md
formulations/<paradigm>.md
env_spec.json
reasoner/<paradigm>_<formulation>.json
builder/<paradigm>_<formulation>_model.py
builder/test_<paradigm>_<formulation>.py
pipeline_state.json
trace.jsonl
\end{verbatim}

El estado del pipeline permite reanudar la ejecución. Las trazas permiten
reconstruir herramientas llamadas, tiempos, errores y artefactos producidos.

\section{Agrex: visualización de trazas de agentes}

Durante el desarrollo se creó Agrex, una librería open source para visualizar
flujos de ejecución de agentes como grafos temporales. La librería está
publicada como visor en:

\begin{verbatim}
https://agrex.ppazosp.dev
\end{verbatim}

y como paquete npm:

\begin{verbatim}
https://www.npmjs.com/package/@ppazosp/agrex
\end{verbatim}

Agrex define un formato de eventos para nodos, aristas, estados, etapas,
marcadores temporales y metadatos. La fase 1 usa el paquete Python
\texttt{agrex} para escribir la traza \texttt{trace.jsonl} durante cada
ejecución. El servidor expone esa traza al frontend, y la interfaz web usa
\texttt{@ppazosp/agrex} para reproducirla como un grafo interactivo.

Esta herramienta no forma parte del contrato científico del modelo generado,
pero sí de la infraestructura de observabilidad del trabajo. Permitió depurar
ejecuciones largas, revisar qué herramientas había llamado cada agente,
identificar errores intermedios y explicar el comportamiento del pipeline con
diagramas de flujo reales en lugar de limitarse a logs textuales.

\section{Ejecución de pruebas}

Las pruebas Python se ejecutan con:

\begin{verbatim}
uv run pytest
\end{verbatim}

Las pruebas de la fase 1 cubren agentes, herramientas, runtime, Router, sistema
de memoria, retrieval, eval harness, servidor y contrato de modelo. Las pruebas
de \texttt{shared/tests/} cubren infraestructura común: settings, storage,
PostgreSQL, Neo4j, Qdrant, embeddings y ciclo de vida de memorias.

\section{Modos de fallo y degradación}

Si PostgreSQL o MinIO fallan, la ejecución no puede completarse correctamente:
son necesarios para metadatos, estado y artefactos. Si Neo4j, Qdrant o los
servicios de embeddings no están disponibles, el sistema puede seguir
generando modelos, pero se desactivan o reducen la recuperación de conocimiento
y la escritura de memoria avanzada.

Los fallos de un subagente no tienen por qué invalidar todo el pipeline. En las
etapas con fan-out, como Formalizer, Reasoner y Builder, el sistema recoge
errores por unidad de trabajo. Esto permite conservar resultados válidos aunque
una formulación o spec concreta falle.

\section{Ejecución desde interfaz web}

Además de la CLI, la fase 1 puede ejecutarse desde la interfaz web incluida en
\texttt{phase1-pablo/web}. En desarrollo local se usan tres piezas: la
infraestructura compartida, el servidor FastAPI de fase 1 y el frontend Vite.

Primero se levantan los servicios compartidos desde la raíz del repositorio:

\begin{verbatim}
docker compose up -d postgres minio neo4j qdrant minio-init
\end{verbatim}

Después, en una terminal, se arranca el servidor de fase 1. Para desarrollo con
el frontend local se usa el puerto \texttt{8000}, porque Vite proxya ahí las
rutas \texttt{/ws} y \texttt{/api}:

\begin{verbatim}
cd phase1-pablo
uv run uvicorn decisionlab.server:app --host 0.0.0.0 --port 8000
\end{verbatim}

En otra terminal se arranca el frontend:

\begin{verbatim}
cd phase1-pablo/web
pnpm install
pnpm dev
\end{verbatim}

La aplicación queda disponible en:

\begin{verbatim}
http://localhost:5173
\end{verbatim}

Desde la interfaz web se puede iniciar un problema de decisión, seguir el estado
de los agentes, revisar artefactos intermedios mediante WebSocket y responder a
los puntos \textit{human-in-the-loop} del pipeline.
