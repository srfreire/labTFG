\chapter{Desarrollo del trabajo}

\section{Metodología de desarrollo con agentes}

\subsection{Metodología general}

El desarrollo de este sistema no siguió un flujo tradicional basado
principalmente en programación manual. Se utilizó una metodología apoyada en
agentes de programación, concretamente Claude Code y Codex, con supervisión
humana en las decisiones de diseño. Esta forma de trabajo se relaciona con
líneas recientes sobre agentes de software, síntesis de programas y desarrollo
multiagente
\cite{chen2021codex,austin2021programsynthesis,yang2024sweagent,qian2024chatdev}.

La idea central fue separar dos tipos de trabajo:

\begin{itemize}
\item las decisiones de arquitectura, alcance y lógica de negocio, que requieren
criterio humano y contexto global del proyecto;
\item la implementación, refactorización, pruebas y corrección de errores, que
pueden delegarse en modelos de lenguaje de alto rendimiento.
\end{itemize}

En tareas acotadas de programación, los modelos actuales muestran un rendimiento
competitivo y permiten acelerar implementación, pruebas y refactorización
\cite{chen2021codex,yang2024sweagent}. Sin embargo, todavía no son igual de
fiables tomando decisiones de diseño sin un contexto completo. Por ello, el
flujo empleado sitúa al humano como \textit{human-in-the-loop} para decisiones
de diseño y a los agentes como ejecutores principales de las tareas de
programación.

La delegación no eliminó la autoría ni la revisión humana. Cada cambio relevante
se trató como una tarea de ingeniería: primero se definía el objetivo, después se
delegaba la ejecución y finalmente se revisaban las evidencias producidas.

Una parte importante de esta metodología fue gestionar la ventana de contexto.
En tareas largas, un agente que acumula demasiados detalles de implementación,
logs, errores intermedios y decisiones locales degrada su capacidad para
mantener una visión global. Por eso el desarrollo no se organizó como una única
conversación larga, sino como un sistema de orquestación: un agente coordinador
mantiene el estado de alto nivel lo más limpio posible y delega tareas cerradas
en agentes trabajadores con contexto propio.

\subsection{Planificación con Linear}

Al tratarse de un proyecto desarrollado por dos personas, una parte importante
del trabajo fue planificar correctamente las tareas compartidas, especialmente
la infraestructura común. Para ello se creó un proyecto en Linear.

El flujo fue el siguiente:

\begin{figure}[H]
\centering
\includegraphics[width=0.68\textwidth]{figuras/phase1-planning-activity}
\caption{Flujo de planificación y ejecución del desarrollo.}
\end{figure}

Primero se iteró con Codex sobre la especificación técnica y el diseño hasta que
el alcance del sistema quedó suficientemente cubierto. Después se dividió el
trabajo en fases. Cada fase tenía criterios de aceptación, y cada tarea dentro
de una fase también tenía sus propios criterios. En Linear, además de la
descripción funcional, cada tarea indicaba dependencias, prioridad, posible
solapamiento con otras tareas y si podía ejecutarse en paralelo.

Estos criterios fueron esenciales porque los agentes necesitaban una forma
objetiva de comprobar si su trabajo estaba terminado. Sin criterios explícitos,
un agente puede generar código funcional a primera vista pero incompleto desde
el punto de vista del producto o de la arquitectura. Por eso se combinaron
criterios de aceptación, validación estructural y pruebas ejecutables, una
decisión coherente con la evaluación funcional de código generado y con enfoques
de generación restringida \cite{chen2021codex,scholak2021picard}.

\subsection{División del trabajo entre agentes}

El proyecto utilizó agentes distintos para tipos de tareas distintos:

\begin{itemize}
\item Codex se empleó principalmente para backend, infraestructura, persistencia,
pruebas y razonamiento técnico sobre el repositorio.
\item Claude Code se empleó principalmente para tareas de frontend y experiencia
de usuario.
\item Skills propias, como \texttt{hydra}, permitieron ejecutar fases completas
de forma autónoma, diferenciando entre tareas de nueva funcionalidad y tareas de
corrección de errores.
\end{itemize}

La división no fue arbitraria. Backend y frontend tienen ritmos de validación
distintos. En backend era especialmente importante respetar contratos,
persistencia, pruebas, migraciones y límites entre fases. En frontend era
importante mantener coherencia visual, ergonomía, interacción y sensibilidad al
detalle de la interfaz. Esta asignación se apoyó tanto en observación práctica
del proyecto como en señales públicas de evaluación: Artificial Analysis separa
capacidades de agentes, programación, razonamiento general y razonamiento
científico, mientras que Code Arena mantiene rankings específicos de desarrollo
web con tareas que exigen razonamiento multi-step y uso de herramientas
\cite{artificialanalysis2026index,lmarena2026webdev}. Estas métricas no se
tomaron como una verdad absoluta, porque cambian con rapidez y dependen del
benchmark, pero sí como una señal de que la elección del modelo debe depender
del tipo de tarea.

\subsection{Orquestación autónoma con Hydra}

La forma más avanzada de trabajo fue la ejecución de fases completas mediante la
skill \texttt{hydra}. Su función no era sustituir a Linear ni a la revisión
humana, sino convertir una lista de tareas ya especificadas en oleadas de
ejecución autónoma. El patrón se parece al propio sistema construido en la fase
1: un orquestador conserva el estado global, delega unidades de trabajo
independientes, espera evidencias, recoge resultados y decide si puede continuar
o si necesita intervención humana.

\begin{figure}[H]
\centering
\includegraphics[width=\textwidth,height=0.80\textheight,keepaspectratio]{figuras/phase1-hydra-activity}
\caption{Flujo de orquestación autónoma con Hydra.}
\end{figure}

El punto clave es que \texttt{hydra} mantiene su propia ventana de contexto lo
más pequeña posible. No carga todos los documentos, diffs y logs de cada tarea.
Lee la información mínima de coordinación, como identificador, estado,
dependencias, tipo de tarea y grupo de solapamiento. El contenido profundo lo
lee cada agente trabajador dentro de su propia sesión. Del mismo modo, cuando
un trabajador termina, el orquestador no absorbe toda la terminal: recibe un
resumen del resultado, el estado del merge, los ficheros tocados y cualquier
nota que requiera seguimiento.

La planificación de cada oleada depende de las dependencias entre tareas. Si una
tarea bloquea a otra, se ejecuta primero la tarea bloqueante. Si dos tareas
tocan zonas relacionadas del sistema, Linear las marca como potencialmente
conflictivas y el orquestador las ejecuta de forma secuencial. Si no se van a
pisar entre sí, puede lanzar varios agentes en paralelo. Cada agente trabaja en
un \textit{worktree} independiente, por lo que sus cambios no contaminan el
árbol principal mientras está desarrollando.

\begin{table}[H]
\begin{center}
\small
\begin{tabular}{|p{3.1cm}|p{4.5cm}|p{4.7cm}|} \hline
\textbf{Paso} & \textbf{Responsable} & \textbf{Evidencia esperada} \\ \hline
Descubrir tareas & Hydra leyendo Linear & Lista de tareas pendientes, dependencias, prioridad y posibilidad de paralelización. \\ \hline
Planificar oleada & Hydra & Conjunto de tareas ejecutables sin blockers activos ni solapamientos peligrosos. \\ \hline
Asignar modelo & Hydra + criterio humano previo & Codex para backend, infraestructura, persistencia y tests; Claude Code para frontend e interacción. \\ \hline
Ejecutar tarea & Agente trabajador & Worktree propio, implementación, pruebas locales y PR hacia \texttt{main}. \\ \hline
Revisar salida & Agente trabajador antes de mergear & Revisión de bugs, silent failures, fallos de arquitectura y convenciones del código. \\ \hline
Cerrar o crear seguimiento & Hydra & PR mergeada, nota de follow-up, nueva tarea para agente o decisión humana pendiente. \\ \hline
\end{tabular}
\caption{Fases del flujo autónomo de desarrollo.}
\end{center}
\end{table}

La revisión final de cada agente se estructuró en cuatro pasos. Primero, buscar
bugs funcionales evidentes. Segundo, actuar como \textit{silent failure hunter}:
detectar casos donde el sistema parece funcionar pero pierde datos, oculta
errores o acepta estados incoherentes. Tercero, revisar fallos de arquitectura,
como acoplamientos indebidos entre fase 1 y fase 2 o responsabilidades mal
colocadas. Cuarto, comprobar convenciones del repositorio, estilo, tests y
contratos locales.

Cuando una tarea terminaba, el agente abría o actualizaba una PR hacia
\texttt{main}. El orquestador monitorizaba las terminales de los agentes y, al
detectar finalización, comprobaba que la PR se hubiese mergeado. Si la PR dejaba
notas, problemas no resueltos o follow-ups, el orquestador decidía si crear una
nueva tarea para otro agente o si marcar el punto como decisión humana. El flujo
continuaba hasta que todas las tareas de la fase quedaban cerradas o hasta que
el sistema se bloqueaba por una decisión que no debía resolver un agente.

Este patrón fue necesario porque las tareas largas de desarrollo autónomo siguen
siendo difíciles. Benchmarks recientes como Terminal-Bench miden tareas
terminales largas con verificación por tests y muestran que incluso agentes
frontera siguen fallando en una parte relevante de escenarios realistas
\cite{merrill2026terminalbench}. Por ello, la metodología del proyecto no
consistió en confiar ciegamente en un agente, sino en aislar tareas, exigir
evidencia ejecutable, revisar antes de mergear y mantener un humano responsable
de las decisiones de producto y arquitectura.

\subsection{Decisiones técnicas y tradeoffs}

Varias decisiones de implementación no fueron delegadas a los agentes, sino
tomadas como restricciones de arquitectura del proyecto. La más importante fue
mantener un Router determinista en lugar de dejar que un agente conversacional
decidiese libremente la siguiente etapa. Esta opción redujo flexibilidad, porque
cada transición debía estar modelada explícitamente, pero mejoró trazabilidad,
reanudación y depuración: ante un fallo era posible saber en qué etapa estaba la
ejecución, qué artefacto faltaba y qué decisión humana bloqueaba el avance. La
misma lógica explica la persistencia temprana en MinIO y PostgreSQL. Guardar
informes, specs, trazas y resultados en cada fase añade complejidad operativa,
pero evita que el estado real del sistema dependa de la ventana de contexto de
un modelo.

Otra decisión deliberada fue separar el contrato externo del modelo generado de
la implementación interna del laboratorio. Se eligió un contrato pequeño por
duck typing, \texttt{decide}, \texttt{update} y \texttt{get\_state}, en vez de
obligar a heredar de una clase del simulador. El tradeoff es que se pierden
algunas garantías estáticas y hay que cubrir más casos mediante tests, pero se
gana desacoplamiento entre fases y facilidad para ejecutar modelos generados
como ficheros Python autocontenidos. También se decidió que la memoria
persistente no se escribiese automáticamente tras cualquier salida del agente:
solo los artefactos aceptados por revisión humana podían consolidarse. Esta
restricción ralentiza el flujo, pero evita que errores científicos o
alucinaciones pasen a formar parte del conocimiento reutilizable del sistema.

\subsection{Criterios de aceptación}

Cada tarea se redactó con criterios de aceptación concretos. Un criterio de
aceptación debía indicar qué comportamiento debía existir, cómo validarlo y qué
evidencia debía quedar.

Ejemplos de criterios usados en este proyecto fueron:

\begin{itemize}
\item una etapa del pipeline debe persistir sus artefactos en una ruta estable;
\item una herramienta debe devolver errores estructurados;
\item una memoria debe escribirse solo tras revisión aceptada;
\item un modelo generado debe pasar sus pruebas con \texttt{uv run pytest};
\item una pantalla debe mostrar estados de ejecución sin romper el layout.
\end{itemize}

Estos criterios permitieron que los agentes trabajasen con autonomía y, al
mismo tiempo, que el resultado fuese verificable.

\section{Supervisión humana y gestión de incertidumbre}

\subsection{Identificación de gaps de diseño}

Un punto importante del flujo fue permitir que los propios agentes identificasen
gaps de diseño. Cuando durante una implementación aparecía una decisión no
cerrada, el agente podía proponer o crear una tarea en Linear para que el equipo
tomase la decisión correspondiente.

Esto evitó que el agente resolviese por su cuenta cuestiones que requerían
criterio humano, como:

\begin{itemize}
\item límites de responsabilidad entre la fase 1 y la fase 2;
\item decisiones de arquitectura compartida;
\item reglas de negocio del pipeline;
\item políticas de memoria, confianza y supersesión;
\item comportamiento esperado ante fallos de infraestructura.
\end{itemize}

De esta forma, los agentes no solo escribieron código, sino que ayudaron a
descubrir incertidumbre técnica. La decisión final, sin embargo, permaneció en
manos humanas.

El flujo de trabajo adoptado puede resumirse así:

Esta metodología influyó directamente en el sistema construido. El pipeline de
generación de modelos también sigue una filosofía similar: los agentes producen
artefactos, pero las decisiones relevantes pasan por revisión humana antes de
quedar aceptadas y almacenadas en memoria. La revisión y las nuevas ejecuciones
funcionan como un mecanismo de refinamiento iterativo guiado por feedback
\cite{madaan2023selfrefine,shinn2023reflexion}.
