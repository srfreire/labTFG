\chapter{Manual de usuario}

Este anexo describe el uso funcional de la fase 1 desde el punto de vista de una
persona que quiere generar modelos de decisión para el laboratorio virtual. No
se asume conocimiento interno de los agentes, aunque sí familiaridad básica con
la ejecución de comandos.

\section{Objetivo del flujo}

El usuario proporciona una descripción del problema de decisión. El sistema
responde con una cadena de artefactos revisables:

El usuario no debe tratar la salida como una respuesta final cerrada. El valor
del sistema está en que cada etapa puede revisarse, corregirse o repetirse.

\section{Inicio de una ejecución}

Una ejecución se inicia con una descripción en lenguaje natural. Conviene que la
descripción indique:

\begin{itemize}
\item qué tipo de decisión se quiere simular;
\item qué contexto o dominio se desea estudiar;
\item si interesa comparar varios paradigmas;
\item qué restricciones o comportamientos esperados son importantes.
\end{itemize}

Ejemplo de entrada:

\begin{verbatim}
Foraging under reward uncertainty.
Compare reinforcement learning and optimal foraging.
The agent moves from local perception and updates from rewards.
\end{verbatim}

\section{Revisión de investigación}

La primera revisión aparece tras la investigación. El sistema propone paradigmas
científicos y conserva informes profundos por paradigma. El usuario debe revisar
si:

\begin{itemize}
\item los paradigmas son relevantes para el problema;
\item falta algún paradigma importante;
\item alguno es demasiado general, demasiado específico o incorrecto;
\item las referencias y postulados son razonables.
\end{itemize}

Solo los paradigmas aprobados continúan. Si falta uno, se puede pedir al sistema
que amplíe la investigación antes de formalizar.

\section{Revisión de formulaciones}

Para cada paradigma aprobado, el sistema genera varias formulaciones
matemáticas. El usuario debe escoger las que quiere convertir en modelos
ejecutables. Para decidir, conviene mirar:

\begin{itemize}
\item variables internas propuestas;
\item parámetros necesarios;
\item reglas de actualización;
\item facilidad de adaptación al entorno;
\item diferencia real entre formulaciones alternativas.
\end{itemize}

No todas las formulaciones generadas tienen que implementarse. La revisión sirve
para evitar que variantes débiles o redundantes pasen a Reasoner.

\section{Especificación del entorno}

Antes de generar specs implementables, el sistema necesita una especificación
del entorno. Esta especificación define acciones, percepción, recursos,
recompensas y restricciones. Un ejemplo mínimo tiene esta forma:

\begin{verbatim}
{
  "grid_width": 10,
  "grid_height": 10,
  "actions": ["up", "down", "left", "right", "stay", "eat"],
  "perception_keys": [
    "x", "y", "grid_width", "grid_height",
    "step", "resources", "last_action_result"
  ],
  "resources": {
    "food": {
      "fields": ["x", "y", "amount"]
    }
  }
}
\end{verbatim}

El Reasoner trata este archivo como fuente autoritativa. Si una formulación
necesita una variable no observable, el sistema debe derivarla de la percepción
o declarar la spec inválida.

\section{Revisión de specs del Reasoner}

El Reasoner produce especificaciones JSON que el Builder puede convertir en
código. El usuario debe revisar:

\begin{itemize}
\item que las acciones existan en el entorno;
\item que las variables internas estén definidas;
\item que las reglas de actualización sean concretas;
\item que los comportamientos esperados sean comprobables;
\item que la adaptación no cambie el significado científico de la formulación.
\end{itemize}

Si el problema está en la formulación, conviene volver a Formalizer. Si el
problema está en la adaptación al entorno, basta con repetir Reasoner.

\section{Revisión de modelos generados}

El Builder genera un archivo de modelo y un archivo de pruebas. El modelo debe
implementar:

\begin{lstlisting}[language=Python]
def decide(self, perception: dict) -> Action
def update(self, action, reward, new_perception) -> None
def get_state(self) -> dict
\end{lstlisting}

El usuario debe comprobar que:

\begin{itemize}
\item \texttt{decide} elige acciones sin mutar estado;
\item \texttt{update} concentra los cambios internos tras recibir recompensa;
\item \texttt{get\_state} expone variables interpretables;
\item las pruebas generadas cubren los comportamientos esperados;
\item el modelo no depende de librerías externas ni de clases internas del simulador.
\end{itemize}

Si el modelo falla tests o no respeta el contrato, se puede repetir la etapa
Builder. Si la causa es una mala spec, se debe volver a Reasoner.

\section{Uso de la memoria}

Cuando la memoria está disponible, los agentes pueden consultar conocimiento de
ejecuciones previas mediante \texttt{retrieve\_knowledge}. El usuario no tiene
que invocar esta herramienta manualmente. Su papel es revisar qué artefactos se
aceptan, porque solo los artefactos aprobados pasan a memoria.

Este diseño evita que un borrador rechazado condicione ejecuciones futuras. La
memoria debe entenderse como una base de conocimiento curada por revisión, no
como un historial completo de todas las respuestas de los modelos.

\section{Resultado final esperado}

Una ejecución satisfactoria produce al menos un modelo Python autocontenido y
sus pruebas. Ese modelo queda preparado para ser usado por el laboratorio
virtual siempre que implemente el contrato \texttt{DecisionModel}. El modelo
final no debe interpretarse de forma aislada: su valor está unido al informe,
la formulación, la spec, las pruebas y las trazas que explican cómo se obtuvo.
