Problema
De literatura a simulación

El cuello de botella no es simular: es convertir conocimiento científico en modelos ejecutables.

Buscar papers, elegir una formulación, adaptarla al entorno y escribir código correcto sigue siendo una tarea manual y difícil de auditar.

Transformación manual clásica

riesgo
Texto científico

Paradigmas, variables, hipótesis y ecuaciones repartidas en literatura.

Interpretación humana

Decisiones no siempre trazables ni reutilizables en otro experimento.

Código ad hoc

La implementación final puede separarse de la formulación original.

Tesis del sistema
Control antes que magia

La solución es un pipeline trazable, no un agente monolítico.

Cada etapa reduce un tipo de incertidumbre y deja un artefacto revisable antes de continuar.

Agentes especializados
Researcher → Formalizer → Reasoner → Builder

Cada agente tiene una frontera cognitiva estrecha.

Researcher

Descubre paradigmas, consulta memoria y lanza investigación profunda.

Formalizer

Convierte informes científicos en formulaciones matemáticas comparables.

Reasoner

Adapta una formulación al entorno y produce una especificación JSON.

Builder

Genera Python autocontenido, tests y validación ejecutable con pytest.

deep/{paradigm}.md → formulations/{slug}.md → reasoner/{formulation}.json → builder/{formulation}_model.py
Arquitectura
Persistencia y trazabilidad

Los artefactos son la unidad de verdad entre etapas.

El sistema no pasa conocimiento crítico solo por prompt: lo escribe, lo versiona por ejecución y lo puede reabrir para revisar o reanudar.

La recuperación combina señales complementarias.

Densa para similitud semántica, BM25 para términos exactos, grafo para relaciones, RRF para fusionar y CRAG para corregir cuando hay duda.

Salida ejecutable
DecisionModel contract

La salida es código cargable por el laboratorio, no un informe.

El contrato es pequeño para que los modelos sean autocontenidos y puedan cargarse por duck typing.

decideelige acción sin mutar estado
updateaprende tras recompensa
stateexpone variables internas
perception = env.build_perception(agent) action = model.decide(perception) reward, result = env.apply(action) new = env.build_perception(agent) model.update(action, reward, new) state = model.get_state()
