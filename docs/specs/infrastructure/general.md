# Infrastructure ‚Äî General Specification

> Status: current | Created: 2026-04-11 | Last updated: 2026-04-11

## Overview

Replace the current file-based communication between Phase 1 (agent pipeline) and Phase 2 (virtual lab) with production-grade infrastructure. Both phases currently rely on hardcoded filesystem paths to exchange artifacts (markdown reports, Python model files, JSON specs, PDFs, charts). A SQLite database in the `shared/` package stores experiment metadata with large JSON blobs inline.

This project introduces MinIO (S3-compatible object storage) for all artifacts, Postgres for structured data, and docker-compose to orchestrate everything. The `shared` package becomes the infrastructure layer that both phases depend on.

## Core Features

- **StorageService** ‚Äî Single MinIO/S3 client in `shared.storage` used by both phases for all file I/O. One bucket (`labtfg`) with prefix-based organization. Replaces all `Path.write_text()` / `Path.read_text()` calls.
- **DatabaseService** ‚Äî Async Postgres via SQLAlchemy + asyncpg in `shared.database`, replacing raw SQLite. Schema managed with Alembic autogenerated migrations.
- **Run ID concept** ‚Äî Each Phase 1 pipeline execution gets a unique `run_id`. Artifacts are stored under `research/{run_id}/`, `models/{run_id}/`. Replaces the single `sample-run/` directory.
- **Phase 1 integration** ‚Äî Modify `reports.py` and `files.py` tool functions (~5 functions) to write artifacts via `StorageService` instead of filesystem.
- **Phase 2 integration** ‚Äî Modify `model_loader.py`, `orchestrator.py`, `reporter.py`, and `api.py` to read artifacts via `StorageService` and use async Postgres instead of SQLite.
- **Docker-compose stack** ‚Äî MinIO, Postgres, Phase 1 server, Phase 2 server, Phase 2 web frontend. All services containerized.
- **Shared lifecycle management** ‚Äî `shared.init(settings)` boots MinIO + Postgres connections, health checks, graceful shutdown.

## Out of Scope

- CI/CD pipeline (future project)
- Cloud deployment (AWS/GCP) ‚Äî MinIO on self-hosted server only
- Authentication/authorization between services
- Phase 1 / Phase 2 application logic changes (only I/O layer changes)
- Frontend changes (web UIs stay the same, only backend data sources change)

## Data Model

### Postgres Tables (replaces SQLite)

**`models`** ‚Äî Phase 1 decision model registry
- `formulation_id` (PK), `class_name`, `paradigm`, `description`, `run_id` (FK), `s3_model_key`, `s3_test_key`, `registered_at`, `metadata` (JSONB)

**`experiments`** ‚Äî Phase 2 experiment lifecycle
- `id` (UUID PK), `created_at`, `updated_at`, `description`, `status` (enum: created/simulated/tracked/analyzed/reported), `spec` (JSONB), `models_used` (JSONB array), `steps`, `seed`, `s3_events_key`, `s3_replay_key`, `s3_tracker_key`, `s3_analyst_key`, `s3_pdf_key`, `s3_tex_key`, `s3_charts_prefix`

**`runs`** (new) ‚Äî Phase 1 pipeline executions
- `id` (UUID PK), `created_at`, `problem_description`, `status`, `s3_report_key`, `s3_prefix`

**`artifacts`** (new) ‚Äî Index of all S3 objects
- `id` (UUID PK), `s3_key` (unique), `artifact_type` (enum: report/deep_report/formulation/reasoner_spec/model/test/pdf/tex/chart/events/replay/tracker/analyst), `run_id` (nullable FK), `experiment_id` (nullable FK), `created_at`, `size_bytes`, `content_type`

### MinIO Bucket Structure

```
labtfg/
  research/{run_id}/report.md
  research/{run_id}/deep/{paradigm}.md
  research/{run_id}/formulations/{paradigm}.md
  models/{run_id}/reasoner/{formulation_id}.json
  models/{run_id}/builder/{formulation_id}_model.py
  models/{run_id}/builder/test_{formulation_id}.py
  experiments/{experiment_id}/events.json
  experiments/{experiment_id}/replay.json
  experiments/{experiment_id}/tracker.json
  experiments/{experiment_id}/analyst.json
  experiments/{experiment_id}/report.pdf
  experiments/{experiment_id}/report.tex
  experiments/{experiment_id}/charts/chart_{n}.png
```

## Integrations

- **MinIO** ‚Äî S3-compatible object storage, runs in Docker, accessed via `minio` Python SDK
- **Postgres** ‚Äî Relational database, runs in Docker, accessed via `asyncpg` + SQLAlchemy async
- **Alembic** ‚Äî Schema migrations, autogenerated from SQLAlchemy models
- **Docker / docker-compose** ‚Äî Container orchestration for all services

## User Flows

### Phase 1 Pipeline Run
1. User starts pipeline via CLI or web: `decisionlab run "food intake"`
2. Pipeline creates a `run` record in Postgres, gets `run_id`
3. Researcher writes `report.md` ‚Üí `StorageService.put("research/{run_id}/report.md", ...)`
4. DeepResearcher writes per-paradigm reports ‚Üí `StorageService.put("research/{run_id}/deep/{paradigm}.md", ...)`
5. Formalizer writes formulations ‚Üí `StorageService.put("research/{run_id}/formulations/{paradigm}.md", ...)`
6. Reasoner writes specs ‚Üí `StorageService.put("models/{run_id}/reasoner/{fid}.json", ...)`
7. Builder writes models ‚Üí `StorageService.put("models/{run_id}/builder/{fid}_model.py", ...)`
8. Each artifact is registered in `artifacts` table
9. Models are registered in `models` table with S3 keys

### Phase 2 Experiment
1. Orchestrator lists available models from `models` table (across all runs)
2. User picks models, simulation runs
3. Events/replay/tracker/analyst JSON ‚Üí stored in MinIO, S3 keys saved in `experiments` row
4. Reporter generates PDF/charts ‚Üí stored in MinIO
5. Experiment status progresses through lifecycle

## Constraints & Non-Functional Requirements

- **Async everywhere** ‚Äî Both phases use asyncio; storage and DB must be async-compatible
- **Backward compatibility** ‚Äî Existing `sample-run/` data must be importable (one-time migration script)
- **Dev simplicity** ‚Äî Single `docker-compose up` boots the full stack
- **Data durability** ‚Äî Postgres data and MinIO data on Docker volumes
- **Connection management** ‚Äî Connection pooling for Postgres, single MinIO client instance

## Key Decisions

| Decision | Rationale |
|----------|-----------|
| MinIO everywhere (no local filesystem backend) | Production-ready from day one; same code path in dev and prod |
| One bucket with prefixes (not multiple buckets) | Simpler permission model, single endpoint config |
| `shared` package as infrastructure layer | Already exists, both phases depend on it, natural home |
| SQLAlchemy async + Alembic | Type-safe models, autogenerated migrations, async-native |
| `run_id` concept for Phase 1 | Enables multiple pipeline runs, versioning, comparison |
| Large JSON blobs in MinIO (not Postgres) | Keeps DB lean; events/replay can be multi-MB |
| `artifacts` table as S3 index | Queryable metadata without listing S3 objects |
